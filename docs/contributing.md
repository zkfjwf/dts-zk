# Contributing

该文档借助了开源项目的contributing文档形式，出于个人对开源的兴趣，此文档仅为尝试性编写。

此文档旨在描述本项目小组开发规范，但也欢迎任何人。

## 单元测试

单元测试很重要。但每个人都不喜欢一上来就写测试用例，而是希望马上开始编写代码，查看效果。

我们采用简化版的**Iterative Test First Programming**的方式，如果你对这一概念不了解，可以查看[MIT的软件工程课中的资料](https://web.mit.edu/6.102/www/sp25/classes/02-testing/#iterative_test-first_programming)。我们强烈鼓励：在你发现你编写的代码有任何错误，需要debug的时候，先针对该错误写一个或多个测试用例，然后再开始修改代码，在通过测试后，你需要将测试用例保留，依据错误的类型，你可能需要把该错误相关的信息注释在代码中。该测试用例可以防止你未来再次犯该错误，并增强你编写的代码的可信度。

## 注释与文档

如果你的代码涉及到整个项目的逻辑设计，你需要更新文档。

通过类型标注和合适的命名，通常你可能不想，也懒得写注释。但当你的代码过长，算法或者业务逻辑很复杂，仅靠命名无法准确地传达含义时，你应当添加注释。

## Commit Message规范

作为Beginner，对于Commit Message，我们没有太严苛的规范，但你需要至少做到以下格式：

```
<type>: <subject>
```

你可以选用任何常见，常用（例如feat, fix, docs），甚至是自己想出来的type词（需要简洁明确）。然后你需要清晰简洁的对你的这次commit做出描述。如果你不确定，你可以搜索互联网或询问大模型。
